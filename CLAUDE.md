## Project Overview

Personal blog/website built with 11ty (Eleventy) v3 static site generator using ESM modules throughout. The site focuses on web development content, particularly web components and modern web APIs. Interactive code examples are powered by custom web components published under the `@jschofield` npm scope.

## Development Commands

- `pnpm start` - Build all components, then start 11ty dev server with live reload (localhost:8080)
- `pnpm build` - Build all components + 11ty static site for production (outputs to `_site/`)
- `pnpm build:components` - Build only the web components
- `pnpm dev:components` - Start all component Vite dev servers in parallel
- No test suite exists in this project

## pnpm Workspace

This is a **pnpm workspace** monorepo. The root package is the 11ty blog, and each web component in `component/` is a workspace member.

- **Workspace config**: `pnpm-workspace.yaml` — defines `packages: ['component/*']`
- **Catalog**: Shared dependency versions (`lit`, `vite`, `typescript`, `prettier`) are pinned in `pnpm-workspace.yaml` under `catalog:`. Components reference these with `"lit": "catalog:"` in their `package.json`.
- **Workspace protocol**: The root `package.json` depends on components via `"@jschofield/play-ground": "workspace:*"` etc.

### Adding a new component

1. Create `component/<name>/` with `package.json`, `tsconfig.json`, `vite.config.ts`, and `src/`
2. Name it `@jschofield/<name>` in `package.json`, use `catalog:` for shared deps
3. Add `"@jschofield/<name>": "workspace:*"` to root `package.json` dependencies
4. Add `import '@jschofield/<name>'` to `site-entry.js`
5. Run `pnpm install` from root

## Build & Deployment

- **Build tool**: 11ty v3 with ESM configuration (`.eleventy.js`)
- **Input**: `src/` → **Output**: `_site/`
- **JS bundling**: `site-entry.js` is the client JS entry point. It's bundled with **esbuild** via an `eleventy.before` hook in `.eleventy.js`, outputting to `_site/index.js`. This resolves bare workspace package imports (`@jschofield/*`) and `baseline-status` into a single bundle.
- **Deployed on Netlify** (`netlify.toml`): production build uses `pnpm build`
- **Netlify functions**: `netlify/functions/` contains `reading-list.mjs` (Google Sheets proxy for `<reading-list>`) and `link-meta.mjs` (OG/Twitter meta-tag proxy for `<link-preview>`, with origin allowlisting and SSRF protection)
- Passthrough copy handles static assets (CSS, JS in posts, images, fonts, videos, webmanifest)

## Architecture

## Changesets & Publishing

Components use **changesets** for versioning and npm publishing.

- **Pre-commit hook** (`.githooks/pre-commit`): Blocks commits that modify `component/*/src/**` without a changeset file. Skip with `--no-verify` for changes that don't affect consumers.
- **Adding a changeset**: Run `pnpm changeset`, select affected packages, choose semver bump, write summary. Then `git add .changeset`.
- **Versioning**: `pnpm changeset version` — bumps versions and writes changelogs based on pending changesets.
- **Publishing**: `pnpm changeset publish` — publishes changed packages to npm under `@jschofield` scope.
- **First-time publish**: `pnpm build:components && pnpm -r --filter './component/*' publish --access public`
- **Config**: `.changeset/config.json` — set to `"access": "public"`, base branch `main`.

## Performance Patterns

When adding new resources or features, maintain these existing patterns:
- **Preloading**: Critical CSS, fonts, and JS are preloaded in `index.njk`
- **Font display**: `font-display: swap` on custom fonts (Atkinson Hyperlegible)
- **Lazy loading**: All post images use `loading="lazy"`; homepage profile uses `fetchpriority="high"`
- **View transitions**: CSS `view-transition: auto` is enabled

## Theming (light/dark)

The site supports three theme states, controlled by `<theme-toggle>` (`src/js/theme-toggle.js`):

- `data-theme="light"` on `<html>` — explicit light
- `data-theme="dark"` on `<html>` — explicit dark
- *(attribute absent)* — "system" mode, follows `prefers-color-scheme`

The user's choice is persisted in `localStorage` under `theme-preference` and reapplied on load.

### The gating pattern (use this everywhere)

Dark mode applies when **the user explicitly chose dark, OR the OS prefers dark and the user has not explicitly chosen light**. In CSS this is two rules:

```css
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    /* dark tokens */
  }
}
:root[data-theme="dark"] {
  /* same dark tokens */
}
```

The `:not([data-theme="light"])` is essential — without it, the OS-dark rule fires even when the user picked light, creating a mismatch between the chrome and dark-mode-aware components.

### Shadow DOM components

Web components with their own dark-mode CSS (e.g. `<javascript-reactivity-quadrants>`) cannot read `prefers-color-scheme` independently of the host's choice. They must use `:host-context()` to mirror the gating:

```css
@media (prefers-color-scheme: dark) {
  :host-context(:root:not([data-theme="light"])) {
    /* dark tokens */
  }
}
:host-context([data-theme="dark"]) {
  /* same dark tokens */
}
```

If a Shadow-DOM component only uses `@media (prefers-color-scheme: dark)` and ignores `data-theme`, it will go dark whenever the OS is dark even if the user toggled the site to light. Always pair the media query with `:host-context()`.

### Adding new dark-mode-aware styles

Prefer adding tokens to `:root` and consuming them via `var(--token)` so the gating only needs to live in one place. Component-specific dark overrides should still follow the two-rule pattern above.
