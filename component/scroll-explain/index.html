The subscriber list

We start with the simplest possible reactive primitive: an array of callback functions and a private variable to hold our data. This is the entire foundation of pub/sub reactivity.

A reactive getter

The store object exposes a value getter that returns the current data. Nothing fancy here — just a clean read interface that hides the private _data variable.

The magic: a reactive setter

This is where it gets interesting. When you set the value, it updates the internal state and then notifies every subscriber. This is the core of reactivity — changes propagate automatically.

Subscribing to changes

The effect method lets anyone register a callback. Push a function onto the subscriber list, and it'll be called every time the value changes. Simple, composable, powerful.

The component

Now we build a web component that uses the store. When the component connects to the DOM, it renders itself and wires up an input listener.

Register the custom element

Finally, register the component with the browser so it can be used as <my-component> in HTML.

Hooking into reactivity

This one line is the key: myStore.effect(() => this.update()). The component subscribes to the store — whenever the value changes (from any source), this component re-renders. Multiple instances of this component will all stay in sync.

Rendering

The render method writes the current value into a <div> and an <input>. This is the initial render — it sets up the full innerHTML.

Handling input

When the user types, the input handler sets myStore.value — which triggers the setter, which notifies all subscribers, which updates all components. The reactive loop is complete. No framework required.

Surgical updates

The update method surgically patches just the parts that changed — the <div> text and the <input> value — avoiding a full innerHTML replace.
